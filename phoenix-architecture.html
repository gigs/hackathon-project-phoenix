Directory layout

cdl/ ├── schema.yaml # core schema definition ├── customers/ │ ├── klarna.md │ ├── santander.md │ ├── revolut.md │ ├── bbva.md │ └── ... ├── queue/ │ └── pending.json # approval queue └── meta/ └── extensions.yaml # registered extensions

Write Contract

Rule 1: Core frontmatter fields must match schema.yaml types and enums

Rule 2: New sections use ## ext:name prefix — no spec change needed

Rule 3: Every write logged with source + timestamp (git commit metadata)

Rule 4: Core fields validated on write; extension fields pass through

Read API

list_customers() — all customers, frontmatter only

get_customer(name) — full parsed customer record

query(filters) — filter by region, health, vertical, stage, DRI

search(text) — full-text search across notes and issues

get_extensions(name, ext) — read a specific extension section

Change Events

Consumers can subscribe to change events instead of polling. Enables reactive workflows.

customer.updated · field.changed · extension.updated

Schema Versioning

Core schema uses semver. Adding fields = minor bump. Changing/removing = major bump + consumer migration. Extensions are unversioned.

Git-Backed Audit

Every write = git commit with source tag. Queryable, diffable, revertable. The safety net for open write access.

Extension Registry

Optional catalog of known extensions. Helps tool builders discover existing data before adding their own. Not enforced.

Validation

Core field validation on every write. Extensions pass through unchecked. Can run as pre-commit hook or CI step.

## Deals

Markdown table. One row per deal/market. deal_id: string # HubSpot deal ID market: string # e.g. "US", "FR" product: string stage: enum[lead,pursuing,...,s4,won,lost,paused] status: enum[green,yellow,red] # from Linear dri: string arr: number forecast: number

## Actuals

Time-series from BigQuery. arr_history: date→number pairs activations: date→number pairs active_subs_history: date→number pairs churn: date→number pairs

## Notes & Issues

Two input paths: Tagged Linear issues: auto-synced, clickable DRI free-text notes: manually written ℹ️ Callouts: time-boxed flags, DRI removes

## Scorecard · ## Changelog

Scorecard: weekly compiled summary (by Claude) Changelog: timestamped entries format: [date] [source] description

## ext:* convention

Freeform sections prefixed with ## ext:. Any tool can create new ones. Documented in extension registry but not enforced.

Example extensions: ## ext:gtm-tracker Tilly's GTM milestones and dates ## ext:meeting-briefs Auto-generated pre-call briefs ## ext:account-overview Louisa's automated summaries ## ext:linear-automation Max's Linear sync metadata ## ext:churn-signals Future: ML-detected risk indicators

Evolution rules

Adding core field → minor version bump Changing core field → major version + migration Removing core field → major version + migration Adding extension → no version change Changing extension → owner's responsibility Removing extension → owner's responsibility Consumers check schema_version in frontmatter to know what to expect.
